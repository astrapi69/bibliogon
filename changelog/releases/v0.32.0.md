# Bibliogon v0.32.0 — 2026-05-14

The "UX-polish + safety net" release: three findings from a manual smoke test of v0.31.0 land as user-facing improvements, plus a batch of security / dependency / infra hardening that accumulated between v0.31.0 and today. 21 commits since v0.31.0.

## Before you install

Bibliogon runs in Docker. You need Docker Desktop installed and running before starting the launcher.

- [Docker installation guide (English)](https://github.com/astrapi69/bibliogon/blob/main/docs/help/en/install/docker-desktop.md) — includes a "Is Docker safe to install?" section
- [Docker-Installationsanleitung (Deutsch)](https://github.com/astrapi69/bibliogon/blob/main/docs/help/de/install/docker-desktop.md) — mit Abschnitt "Ist Docker sicher zu installieren?"

The launcher detects Docker, downloads Bibliogon automatically, and opens it in your browser.

## Action required

None. v0.32.0 is a minor bump that adds new endpoints + UI surface without changing any v0.31.0 schema. The reclassify endpoints are additive; existing import flows are unchanged.

## What's new

**Article ⇄ Comment reclassify** — when the comment-detection heuristic misclassifies in either direction, fix it in one click. ArticleEditor's header now has a `(...)` kebab menu with a "Move to comments" item; the Settings → Comments admin tab has a "Move to articles" button next to each row's trash icon. Both moves are transactional (the source row is deleted in the same commit as the destination row is inserted, never a half-state) and reversible via the reciprocal direction, so a simple confirm dialog gates each one — the type-to-confirm pattern from bulk-delete would be wasted friction. The deep-link toast that fires after a move offers a one-click "Open Comments admin" / "View article" button so the user can verify the move landed.

**Better comment detection in the Medium importer.** v0.31.0's strict "body < 500 chars AND no structural elements" rule missed comment-shaped replies that crossed the 500-char gate (one real-world example: a 941-char reply opening with "This is a powerful and unsettling reframing..." and closing with a question — clearly a comment, but the length pushed it past the gate). v0.32.0 adds a tier-2 rule that catches longer comments when (a) no article-shape disqualifiers (no headings, no code blocks, no images, body_len < 2000) AND (b) at least one conversational signal: second-person opener (Your/You/Du/Dein/Deine/Ihre), opening question, or closing question. Data-validated against the 209-file production Medium export: 3 new true-positive detections, 0 false positives, 0 lost detections. An earlier draft of the rule that scored 3+ generic signals (without requiring a conversational marker) flagged a German image-poem as a comment; the conversational-marker gate excludes that false positive while still catching the user-reported edge case. Audit at `docs/audits/medium-comment-heuristic-2026-05-14.md`.

**Editor "Copy" split-button.** Toolbar gets a new Copy button that writes the article / chapter body to the clipboard. Primary click copies as Markdown (the default, since Markdown preserves headings + lists + links + images for paste-targets that render it); a chevron disclosure exposes "Copy as plain text" for paste-targets that mangle Markdown (email, notes, chat). Both modes prepend the document title (+ optional subtitle for articles) so the paste-target keeps the context, not just the body. Plain-text rendering keeps lists and blockquote prefixes, surfaces link references as `text (url)` so they survive the paste, and substitutes images with `[Image: alt — caption]`. The TipTap-to-Markdown converter was extracted from Editor.tsx into a new shared module so the same code powers both the existing WYSIWYG ↔ Markdown toggle and the Copy action.

**Medium-import Start button fix.** Smoke-test finding from v0.31.0: after a successful import, the Start button stayed enabled with the same ZIP still selected, so an accidental second click triggered a re-import. (The backend deduped safely so no data was duplicated — but the UX read as "did it really import?".) The file is now auto-cleared on success; the result panel's "Weiteres ZIP importieren" button covers the back-to-idle path. Failure paths are unchanged: a failed import keeps the file selected so the user can retry without re-picking.

**Misleading wait-time copy fixed.** The Medium-import progress UI said "this can take up to one minute" in 8 catalogs. The claim was false for any archive over ~50 MB — a 500 MB export takes substantially longer than 60 s, so the hard time bound created a "false-crash" impression. All 8 catalogs now frame the wait as "larger archives may take longer" without a specific time bound.

**500 MB upload cap.** New `BodySizeLimitMiddleware` rejects POST / PUT / PATCH bodies larger than 500 MB before any route handler sees the request, so the worst-case unauthenticated upload can't exhaust memory. Closes the open backend gap noted in v0.31.0's Medium-import upload-zone helper text (the 200 MB frontend guard had no backend twin).

## Fixed

**BookEditor load effect ran twice under React StrictMode**, occasionally landing a stale book on top of a freshly-loaded one. The fix adds a `cancelled` flag captured by the effect's cleanup.

**`backup_history.json` lost entries** when the JobStore wrote from a second worker process. The in-memory list cached the first writer's state; subsequent reads from a second process saw the cached list, not the on-disk file. `list()` and `add()` now reload from disk on every call. Closes `BACKUP-HISTORY-SINGLETON-01`.

## Internal

- **Mutation testing unblocked.** First successful `mutmut` run on the `app/import_plugins/` audit scope produces a **77.8% mutation score** (2156 / 2770 mutants killed). Clears the >= 60% acceptance bar in `.claude/rules/quality-checks.md`. The CI mutation workflow had been wired for 10 days without ever running successfully — the first manual `workflow_dispatch` exposed a `BadTestExecutionCommandsException` that the fix resolves. Now a lessons-learned rule: "Operational gaps masquerade as wired infrastructure".

- **User-overlay write path** for `settings.py` and `plugin-install` resolves writes via `app.paths.get_data_dir()` + a new overlay file. The remaining two CWD-relative writes flagged in v0.31.0's `PROD-WRITES-ARCHITECTURE-01` (P3) ship in this release.

- **Dependency bumps.** Pillow 11.3.0 → 12.2.0 in launcher (CVE batch); cryptography 46.0.7 → 48.0.0 (`CRYPTOGRAPHY-V48-MIGRATION-01` closed).

- **Two new E2E smoke specs.** `e2e/smoke/reclassify.spec.ts` pins the Article → Comment and Comment → Article round-trip. `e2e/smoke/copy-toolbar.spec.ts` pins the Copy split-button surface + the chevron menu.

## Verifying downloads

Each release artifact is published with a SHA-256 hash. Verify before installing on production machines:

```bash
sha256sum <downloaded-file>
```

The expected hash is listed alongside each artifact on the GitHub release page. The `install.sh` one-liner verifies the launcher binary automatically.
