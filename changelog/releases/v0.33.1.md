# Bibliogon v0.33.1 — 2026-05-16

The "Comments-Trash recovery + Authors-Database foundation" point release. Closes a v0.32.0 omission where moved-to-trash comments became invisible, ships the Authors-Database foundation (Wizard datalist work follows in the next release), and adds two new lessons-learned rules ("Half-wired trash lifecycle" + "Test-isolation discipline").

## Heads-up for users who pressed "Move to Trash" on comments before v0.33.1

**English.** If you moved comments to trash on v0.32.0 or v0.33.0 and couldn't find them afterwards, your data was not lost — the row stayed in the database with a "soft-delete" marker, but the Comments-Admin tab in Settings had no surface to show you the trash or restore from it. v0.33.1 adds that surface. After upgrading, open **Settings → Kommentare** (Comments) and click the new **Papierkorb** (Trash) toggle next to the active list — every previously-trashed comment will appear there, with a per-row **Wiederherstellen** (Restore) button and a bulk-restore affordance.

**Deutsch.** Falls Sie in v0.32.0 oder v0.33.0 Kommentare in den Papierkorb verschoben haben und sie danach nicht mehr finden konnten — Ihre Daten sind nicht verloren. Die Zeile blieb mit einer "Soft-Delete"-Markierung in der Datenbank, aber der Comments-Admin-Tab in den Einstellungen hatte keine Oberfläche, um den Papierkorb anzuzeigen oder wiederherzustellen. v0.33.1 ergänzt diese Oberfläche. Nach dem Upgrade öffnen Sie **Einstellungen → Kommentare** und klicken Sie auf den neuen **Papierkorb**-Schalter neben der aktiven Liste — alle zuvor in den Papierkorb verschobenen Kommentare erscheinen dort, mit einem **Wiederherstellen**-Button pro Zeile und einer Bulk-Wiederherstellung.

## Action required

None. v0.33.1 is a patch bump. The new comments-trash endpoints (`GET /api/comments/trash/list`, `POST /api/comments/trash/{id}/restore`, `DELETE /api/comments/trash/{id}`, `DELETE /api/comments/trash/empty`, `POST /api/comments/trash/bulk-restore`) and the Authors-Database endpoints (`/api/authors` CRUD) ship as additive surface; every v0.33.0 flow is unchanged.

The Alembic schema gains one new table (`authors`) for the Authors-Database. The migration is reversible and the table starts empty — no data migration from existing free-text `Book.author` / `Article.author` columns (the Authors-DB is decoupled and opt-in per design decision D5).

## What's new

**Comments trash-lifecycle (Bug 10 fix).** Comments shipped soft-delete in v0.32.0, but the counterparts (list-trashed, restore, permanent-delete-from-trash, empty-trash) were filed as "v2" in the original commit and never picked up. Production smoke surfaced **61 user-trashed comments stuck in invisible purgatory**: the user had pressed "Move to Trash", got a "moved to trash" toast, and then had no surface to look in. v0.33.1 closes the gap with:

- Backend: four new endpoints (`GET /api/comments/trash/list`, `POST /api/comments/trash/{id}/restore`, `DELETE /api/comments/trash/{id}`, `DELETE /api/comments/trash/empty`) + a bulk-restore endpoint (`POST /api/comments/trash/bulk-restore`). Bulk-permanent-delete-from-trash reuses the existing `POST /api/comments/bulk-delete?permanent=true`.
- Frontend: a `viewMode` toggle on Settings → Comments with `comments-active-toggle` and `comments-trash-toggle` buttons. The Trash toggle carries a badge with the trash population count. Per-row actions in trash mode are Restore + Permanent-Delete (replacing the active-view Trash icon). An "Empty trash" CTA in the toggle row + a danger-variant confirm. Bulk affordances (Restore + Permanent-Delete) in trash view via a dedicated `comments-trash-bulk-action-bar`.
- Coverage: 22 new backend pytest cases (16 single + 6 bulk-restore), 22 new frontend Vitest cases (7 in client.test.ts, 15 in CommentsAdminSection.test.tsx), 4 Playwright smoke specs covering the full user-visible flow.

**Authors-Database foundation (Bug 8 Phase 1).** A new standalone `authors` table + `/api/authors` CRUD + a new Settings "Autoren-Datenbank" / "Authors Database" tab sibling to the existing personal-identity "Autor" tab. The Wizard datalist integration that will let the Article-to-Book conversion pre-fill suggested authors lands in the next release as Bug 8 Phase 2; this release ships the storage layer + admin surface. Decoupled from existing `Book.author` / `Article.author` columns by design (no FK, no backfill — they remain free-text and the Authors-DB is an opt-in suggestion layer).

- Schema: `Author(id, name, slug, bio, created_at, updated_at)`. Slug is server-generated from name (lowercase + hyphenated, German + Nordic diacritics transliterated to ASCII before NFKD-fold for other diacritics, all-emoji fallback to `"author"`, collisions appended with `-2 / -3 / ...`).
- Slug is immutable after create — protects any future `/authors/{slug}` URL routing from breaking on rename.
- DELETE is idempotent (204 on already-gone) — supports multi-tab races.
- Coverage: 52 new backend pytest cases, 7 client.test.ts cases, 17 component Vitest cases, 16 i18n keys across all 8 catalogs (DE + EN native, ES / FR / EL / PT / TR / JA passthru-EN per the `REVIEW_STATUS.md` policy).

**Articles-vs-Books parallel-surface asymmetry tally update.** The pattern is now at **8 confirmed instances** (Bug 10 is #8; Bug 4a is #7). Both Bug 10 and Bug 8 ship with the standard "intentional asymmetry documentation rule" (the Authors-Database Wizard-integration intentionally lands Books-only in Phase 2 — Articles get the same surface in a later session; the asymmetry is design-deliberate not a regression).

**Two new lessons-learned rules**:

1. *"Half-wired trash lifecycle: soft-delete shipped without the restore-surface is purgatory, not a feature"*. When a feature ships one half of a soft-delete lifecycle and the other half lands in a "v2 ships" docstring, the deferred half MUST be filed as a load-bearing backlog item with a real ID — not as docstring prose. Comments-trash is the canonical concrete example. Detection grep: `grep -rnE 'out of scope|v2 ships|deferred to v2|TODO.*v2' backend/app/ frontend/src/ plugins/`.

2. *"Test-isolation discipline: never run integration smoke-tests outside pytest"*. The Bibliogon harness's three-layer isolation (`BIBLIOGON_TEST=1` env-var, `:memory:` DB URL, `.bibliogon-production` marker tripwire) only fires under pytest. A bare `poetry run python -c "from app.main import app; ..."` script bypasses every guard and points at the real production DB. Rule: integration smoke-tests are pytest files, not free-standing scripts. Acceptable shortcut: prefix the command with `BIBLIOGON_TEST=1 TEST_DATABASE_URL=sqlite:///:memory:`. Never the bare form.

## For testing this release

The Comments trash-lifecycle ships with 4 Playwright smoke specs at `e2e/smoke/comments-trash-lifecycle.spec.ts`:

1. Single soft-delete → trash view shows row → restore → back in active.
2. Permanent-delete-from-trash via confirm dialog removes row from DB.
3. Bulk-restore returns selected rows to the active list.
4. Bulk-permanent in trash gated by TypeToConfirmDialog hard-deletes the rows.

Run with `npx playwright test --project=smoke comments-trash-lifecycle.spec.ts`. The existing comments-admin-bulk-delete spec covers the active-view flow.

The Authors-Database ships with 52 backend pytest cases + 17 component Vitest cases. No E2E spec yet — Phase 2 of Bug 8 will add a Wizard-datalist E2E spec covering the end-to-end flow.
