# Bibliogon v0.41.0 — 2026-05-30

A UX/UI theme + accessibility hardening release, plus per-content-type field visibility and a documentation refresh. 24 commits since v0.40.0. The headline is a full audit of all **12 theme variants** (6 palettes × light/dark) that corrected a long-standing "5 palettes / 10 variants" miscount, found and fixed a class of undefined-CSS-token bugs the existing completeness hook was structurally blind to, and shipped a new `make verify-theme` release gate.

## Before you install

Bibliogon runs in Docker. You need Docker Desktop installed and running before starting the launcher.

- [Docker installation guide (English)](https://github.com/astrapi69/bibliogon/blob/main/docs/help/en/install/docker-desktop.md) — includes a "Is Docker safe to install?" section
- [Docker-Installationsanleitung (Deutsch)](https://github.com/astrapi69/bibliogon/blob/main/docs/help/de/install/docker-desktop.md) — mit Abschnitt "Ist Docker sicher zu installieren?"

The launcher detects Docker, downloads Bibliogon automatically, and opens it in your browser. The first launch takes 5-10 minutes (Docker images build, ~2 GB disk space).

## Download

| Platform | File |
|----------|------|
| Windows | `bibliogon-launcher.exe` |
| macOS (Apple silicon) | `bibliogon-launcher-macos.zip` |
| Linux | `bibliogon-launcher-linux` (ELF binary) |

Each platform also ships a `*.sha256` checksum next to the binary.

## Verifying downloads

```bash
# macOS / Linux
shasum -a 256 bibliogon-launcher-<platform>
cat bibliogon-launcher-<platform>.sha256
```

```powershell
# Windows
Get-FileHash -Algorithm SHA256 .\bibliogon-launcher.exe
Get-Content .\bibliogon-launcher.exe.sha256
```

The hashes must match.

If your operating system warns about an unsigned binary, see the [Bibliogon installation overview](https://github.com/astrapi69/bibliogon/blob/main/docs/help/en/installation.md).

## What's new

### Per-content-type field visibility

The ArticleEditor sidebar now shows only the fields relevant to the article's content type, instead of every field for every type. A new per-type `core_fields` list in the single source of truth `backend/config/content-types.yaml` drives which optional fields (Tags / Excerpt / SEO / Canonical URL / Featured image) each type renders; identity fields (title, content, status, subtitle, author, language, topic) are always shown. For example a Newsletter shows none of the optional core fields (it has its own issue-number / send-date), a Short story shows only Tags, and Canonical URL is Blog-post-only. `null`/omitted = show all (backward-compatible), `[]` = none; a field-validator rejects unknown field keys.

### Theme + accessibility hardening (12 variants)

A full audit of all 6 palettes × light/dark:

- **`make verify-theme` release gate** (now part of `make release-test`): undefined-token detection (catches bare `var(--token)` references to tokens defined in no palette — the blind spot the old hex-fallback-only hook missed), 96 WCAG-2.1 contrast checks across all 12 variants, and a hardcoded-hex lint.
- **18 undefined CSS tokens** resolved to defined semantic tokens (`--bg`, `--accent-bg`, `--accent-text`, `--error`, `--bg-surface`, `--accent-muted`, and 12 more found by the new gate). These rendered nothing and were invisible to UI tests; most visible effects were Storyboard cards and the bulk-action-bar active state not rendering their intended surfaces.
- **31 hardcoded status colors** replaced with semantic `var(--success / --danger / --accent / --warning)` so they flip correctly in dark mode.
- **Dark-mode contrast**: `--text-muted` brightened across all dark palettes, nord dark `--danger` lightened to a legible ratio, and the EditableTitle published-work warning button switched to `--text-inverse` (white-on-light-accent had failed in all 6 dark variants).
- **Comic bubble + tail keyboard operation**: the drag handles now respond to Enter/Space (select) and Arrow keys (reposition) — a keyboard equivalent of dragging.
- **Comic editor chrome** (panels, multi-panel grid, tail-handle ring, picture-book speech-bubble layout) is theme-aware; bubble *defaults* stay comic-convention (mirrored in the PDF walker).
- **Article-Dashboard header** stays single-line at 900px+ (the content-type split-button had pushed it past the wrap threshold; Backup folded into the Import chevron). Playwright regression pin added.
- Storyboard mood-dot ring visibility + collage drag-handle affordance fixes.

### Documentation

- 35 help screenshots regenerated to the current UI.
- The Articles content-types help page gained a comparison table, a field-visibility matrix, and a per-type detail section for all 8 types (DE + EN).
- The "5 palettes / 10 variants" miscount corrected to 6 / 12 across README, README-de, CLAUDE.md and the architecture rule (Warm Literary, the default palette, was never counted).
- New developer theming guide (`docs/development/theming.md`).
- New `verify_docs_completeness.py` release gate (version headers, help-i18n parity, image + cross-reference integrity).

## Fixed

- Storyboard cards and the article/book bulk-action-bar active state rendered transparent/indistinct because of undefined CSS tokens.
- Status icons/badges rendered the light-mode shade in dark mode (hardcoded hex).
- Muted/secondary text was below WCAG AA on dark cards and soft surfaces.
- The Article-Dashboard header wrapped to two lines at common widths.
- A help-image path error caught by the new docs-completeness gate.

## Compatibility

- No schema migrations, no new columns, no config-format changes.
- `content-types.yaml` gains an optional `core_fields` key per type; omitting it preserves the previous "show all fields" behavior, so custom or future types degrade gracefully.
- Theme behavior is backward-compatible; only token *values* and previously-undefined token *references* changed.
