# Bibliogon v0.24.0

Article authoring lands as a first-class feature alongside book authoring. Articles are standalone documents (not book chapters) intended for blog posts, newsletter pieces, and online publication. The release ships AR-01 (entity + CRUD + editor), AR-02 (publications + drift detection across 8 platforms), AR-02 Phase 2.1 (topics + SEO), and three editor-parity phases that bring articles to feature parity with the book editor: a shared `RichTextEditor` with content-kind-aware plugin gating, per-article ms-tools quality checks, translate-article via the existing translation provider abstraction, and Markdown / HTML / PDF / DOCX export. Plus PS-13 "Save as new chapter" in the chapter conflict dialog, UX-FU-02 featured image upload, follow-ups for PGS-02..04 (per-book PAT credential integration, mark_conflict + rename detection in the smart-merge diff, surface skipped branches in multi-language import), and a refreshed README + UX conventions guide + plugin author patterns.

## Action required

This release adds four Alembic migrations for the new `articles`, `publications`, and `article_assets` tables plus the topic/SEO columns. If you reach v0.24.0 via `alembic upgrade head` rather than fresh install, run it after pulling:

```bash
cd backend && poetry run alembic upgrade head
```

All migrations are idempotent.

## Added

### Article authoring (AR-01 Phase 1)

- **Article entity.** New `Article` model + Alembic migration `f9a0b1c2d3e4`. Standalone document with title, content (TipTap JSON), language, tags, status (`draft` / `ready` / `published` / `archived`), author, timestamps. CRUD endpoints under `/api/articles/`. Soft-delete semantics mirror books.
- **ArticleEditor.** New `frontend/src/pages/ArticleEditor.tsx` mounts the shared editor with article-specific sidebar (status, language, author, tags, SEO, featured image, topic). Dashboard gains a "New Article" entry alongside book creation; article list view at `/articles/`.
- **Status lifecycle with "ready" stage.** Articles flow `draft -> ready -> published -> archived`. The `ready` status (added on top of the original draft/published/archived set) marks an article as approved-for-publish without yet being live.
- **Featured image (URL + upload).** Per-article hero image surfaces in social previews. UX-FU-02 ships the upload widget; existing URL input stays as a free-text alternative.

### Publications + drift detection (AR-02 Phase 2)

- **Publication entity.** New `Publication` model with cascade-delete to articles + Alembic migration `a0b1c2d3e4f5`. Tracks where an article was published: platform name, external URL, published-at timestamp, content snapshot for drift detection.
- **8 platform schemas** as YAML data: Medium, Substack, X, LinkedIn, dev.to, Mastodon, Bluesky, Generic. Each declares its own field set; the frontend renders the form schema-driven so adding a new platform is a YAML edit.
- **Drift detection.** When the article body changes after a publication is recorded, the `Publication` row's snapshot diverges from the live article and the publication appears as `out_of_sync`. Verify-live timestamp tracks last-checked.
- **PublicationsPanel.** Sidebar component shows every recorded publication with platform badge, drift status, and a "Mark as published" / "Verify live" action. AddPublicationModal renders the schema-driven form.
- **Article-level SEO fields:** `canonical_url`, `featured_image_url`, `excerpt`, `tags`. Used as fallbacks for individual publication metadata.

### Topics, SEO, and sidebar layout (AR-02 Phase 2.1)

- **Topics tab in Settings.** New 4th tab (`tab_topics`) manages the topic list applied across articles. Inline add via `<input>` + add button; topics persist to `app.yaml`.
- **Topic dropdown in editor** with inline-add: pick from the configured topic list or type a new value to add it on the fly.
- **SEO Title / SEO Description fields.** Three-row textareas in the article sidebar; persist on blur. Used by PublicationsPanel as the default SEO copy when a platform supports it.
- **Sidebar-left layout.** Article sidebar mirrors the BookEditor layout convention - sidebar on the left, editor body on the right. Smoke-tested for layout-critical viewports.
- **`useTopics` hook.** Reads `/api/settings/app` and exposes the topic array; gracefully falls back to an empty list when the API call fails.

### Editor-Parity Phase 1 (RichTextEditor extraction)

- **`contentKind` prop.** The shared `Editor` component now takes `contentKind: "book-chapter" | "article"` and gates which TipTap extensions + AI prompt set it activates. Books keep the full plugin surface; articles get the article-relevant subset (no chapter-only features).
- **`editor-gates` module.** New `pluginsForContentKind` helper centralises the gating logic so future content kinds drop in one place.
- **Article-specific AI prompts.** When `contentKind === "article"`, the AI rewrite / improve / expand prompts pivot from "your novel chapter" to "your article" wording so suggestions match the genre.

### Editor-Parity Phase 2 (ms-tools per-article + translate-article)

- **ms-tools per-article.** No code change required - the existing `/api/ms-tools/check` endpoint already accepts an optional `book_id`; articles call it with `article_id` instead. Style checks, sanitization, and metrics now apply to articles identically.
- **Translate-article endpoint.** `POST /api/articles/{id}/translate` runs `content_json` through the existing translation provider abstraction (DeepL, LMStudio). Creates a new article in the target language with the translated body, status `draft`, and a recorded link back to the source.
- **Translate sidebar panel.** Mirrors the book translate UI: provider picker, target language picker, translate button, post-translate toast linking to the new article.
- **Provider gating.** Translate dropdown filters out unconfigured + unhealthy providers so the user only sees options that will actually work; the panel shows a config gate when no provider is configured (link to Settings > Plugins > Translation).
- **502 surfacing.** Provider errors propagate as HTTP 502 with the provider name + reason rather than a generic 500, so the frontend toast can route to a specific message.
- **Rebuild fallback + diagnostic logs.** When the translation provider returns malformed output, the endpoint logs the raw response and returns a structured error rather than crashing.

### Editor-Parity Phase 3 (article export)

- **Export to Markdown / HTML / PDF / DOCX.** New router `app/routers/article_export.py` reuses `bibliogon_export.tiptap_to_md.tiptap_to_markdown` for the JSON-to-Markdown conversion + shells out to Pandoc for PDF and DOCX. Markdown and HTML paths are pure-Python and always available.
- **Sidebar Export panel** with one button per format. Disabled state during export; toast on success/failure. ApiError parity with the rest of the app.
- **Frontend download via fetch + Blob** (not a plain link click) so backend errors surface via the standard ApiError toast path.
- **11 backend pytest tests** cover the converter reuse, Pandoc invocation (mocked), happy path per format, error paths, and content-disposition header.

### UX-FU-02 (article featured image upload)

- **Per-article asset uploads.** Mirrors `api.covers` for books: `POST /api/articles/{id}/featured-image` accepts a multipart upload, validates extension + size, stores under `uploads/articles/{id}/`, and returns the URL.
- **`ArticleImageUpload` widget** combines URL input with a drag-and-drop / click-to-upload zone. Uploading or pasting a URL both populate the same field.
- **Migration `c2d3e4f5a6b7`** adds the `article_assets` table. Schema mirrors the book-asset table.

### PS-13 (Save as new chapter)

- **Third option in `ConflictResolutionDialog`.** Alongside Keep Local / Discard, a "Save as new chapter" button forks the unsaved local edit into a fresh chapter inserted at `source.position + 1`; positions of subsequent chapters bump by 1 to keep the list gap-free. Source chapter is left untouched (keeps server content).
- **`POST /api/books/{id}/chapters/{cid}/fork`** with `ChapterFork` schema. Inherits `chapter_type` from source. Returns 201 on success, 404 on missing book/chapter.
- **5 i18n keys × 8 languages:** `save_as_new_chapter`, `save_as_new_chapter_tooltip`, `local_draft_suffix`, `saved_as_new_chapter`, `save_as_new_failed`.
- **6 backend pytest tests + 3 Vitest tests.**

### plugin-git-sync follow-ups

- **PGS-02-FU-01: per-book PAT credential integration.** New shared helper `app/services/git_credentials.py` owns per-book PAT and SSH env. plugin-git-sync push uses the one-shot pushurl pattern; PAT never lands in `.git/config`. Endpoints `PUT/GET/DELETE /api/git-sync/{book_id}/credentials` + a `CredentialsSection` in `GitSyncDialog`. 12 i18n keys × 8 languages. +29 tests (20 helper + 5 endpoint + 4 dialog).
- **PGS-03-FU-01: `mark_conflict` + rename detection.** New resolution action rewrites `both_changed` chapters with git-style conflict markers so the user can keep both versions in-line. `_collapse_renames` post-process step pairs `*_removed` + `*_added` rows with matching bodies into `renamed_remote` / `renamed_local` rows carrying `rename_from`. `take_remote` on rename rows updates DB title only. Counts payload gains `marked` + `renamed`. 6 i18n keys × 8 languages. +9 backend tests + +3 Vitest tests.
- **PGS-04-FU-01: surface skipped branches in multi-language import.** When per-branch import fails (auth, missing metadata, importer error), the response now lists the skipped branches with reason so the frontend can show a result panel. New reusable result panel for skipped branches. +tests.

### Multi-book wizard finishing (CIO-08-FU-01)

- **`useMachine(wizardMachine)`** replaces the parallel `useState<WizardState>` in ImportWizardModal. New `SuccessMultiStep` terminal lists every imported book with per-book "Open" link (no auto-redirect). ExecutingStep `onSuccess` signature gains `bookIds`. 4 i18n keys × 8 languages. +5 Vitest tests.

## Changed

- **README rewritten for v0.23.0** (the EN README was lagging several releases). README-DE refreshed and synced to the same content. Both now reflect articles, git-sync, and multi-book import as first-class features.
