## Before you install

Bibliogon runs in Docker. You need Docker Desktop installed and running before starting the launcher.

- [Docker installation guide (English)](https://github.com/astrapi69/bibliogon/blob/main/docs/help/en/install/docker-desktop.md) - includes a "Is Docker safe to install?" section
- [Docker-Installationsanleitung (Deutsch)](https://github.com/astrapi69/bibliogon/blob/main/docs/help/de/install/docker-desktop.md) - mit Abschnitt "Ist Docker sicher zu installieren?"

The launcher detects Docker, downloads Bibliogon automatically, and opens it in your browser. The first launch takes 5-10 minutes (Docker images build, ~2 GB disk space).

## Download

| Platform | File |
|----------|------|
| Windows | `bibliogon-launcher.exe` |
| macOS (Apple silicon) | `bibliogon-launcher-macos.zip` |
| Linux | `bibliogon-launcher-linux` (ELF binary) |

Each platform also ships a `*.sha256` checksum next to the binary.

## Verifying downloads

```bash
# macOS / Linux
shasum -a 256 bibliogon-launcher-<platform>
cat bibliogon-launcher-<platform>.sha256
```

```powershell
# Windows
Get-FileHash -Algorithm SHA256 .\bibliogon-launcher.exe
Get-Content .\bibliogon-launcher.exe.sha256
```

The hashes must match.

If your operating system warns about an unsigned binary, see the [Bibliogon installation overview](https://github.com/astrapi69/bibliogon/blob/main/docs/help/en/installation.md).

## What's new

**Article-to-Book conversion.** Select multiple articles on the Articles dashboard, click "Als Buch" / "As book" in the bulk-action bar, and walk through a guided 6-step wizard to produce a new book with each article as a chapter. The wizard collects the metadata (title, author, optional subtitle / series / cover image â€” single-article conversions pre-fill subtitle and cover from the source article), lets you add optional front-matter (title page, dedication, introduction) and back-matter (acknowledgments, about-author) chapters, and choose the chapter order (date asc/desc, title asc/desc, or manual drag-reorder). The whole creation is one transactional database write so an interrupted call leaves no half-state. Original articles stay untouched on the dashboard - the new book holds an independent copy of each article's body. After a successful conversion, a "Buch Ă¶ffnen" / "View book" toast CTA navigates to the new book in the Book editor; you can also dismiss the toast and stay on the Articles dashboard. The whole feature is fully documented under `docs/help/{en,de}/articles/convert-to-book.md` plus the smoke-test plan + bilingual manual test guide at `docs/testing/smoke-tests/article-to-book-conversion*.md`. Coverage: 37 backend integration tests + 12 Vitest specs + 3 Playwright smoke specs.

**UX-Full-Audit closures (14 items).** The 2026-05-14/15 UX-Full-Audit identified 23 findings across the Bibliogon UI; this release closes the actionable P3 backlog. Headlines: a shared `<LoadingIndicator>` component (with a fixed global `.spin` animation), a shared `<EmptyState>` component migrated to 6 callsites including a new BookEditor empty-state surface, `ArticleFilterBar` extracted from the inline-in-ArticleList monolith, the Settings monolith split into per-tab components (`AppSettings`, `AiAssistantSettings`, `TopicsSettings`, `PluginSettings`, `AuthorSettings`), full `data-testid` coverage on the BookEditor (was zero pre-audit), view-agnostic `data-{book,article}-id` attributes for E2E targeting across card + list views, 5 theme-token gaps closed across the 111 callsites (with an audit script for future drift), 62 % of `notify.error` catch sites fixed to forward the caught error so the rich Issue-Melden affordance fires (plus a pre-commit hook to prevent regression).

**Restore-button feels-broken finding (was a perception lag, not a code bug).** A user-reported "Articles-Trash restore is broken" investigation surfaced a 419 ms click handler chaining two synchronous network round-trips with no intermediate UI feedback. Fix: optimistic state update + use the response entity directly instead of refetching the whole list. Restore now feels instantaneous. The workbox console message that was misread as "blocking the request" was promoted to a lessons-learned rule ("Workbox 'No route found' is benign info, not a bug indicator") so the next investigation doesn't conflate the SW log with a functional break.

**BulkActionBar stale-state fix.** Single-item destructive handlers (soft-delete, permanent-delete, restore from trash) now reconcile the bulk-selection set after the API call succeeds, so the bar's count never references an orphan id. 14 Vitest hook tests pin the behavior for both `useArticleSelection` and `useBookSelection`; an E2E backfill covers the user-visible path.

**Test isolation hardening.** A 5-test cross-file regression caused by a stale `platform_schema` LRU cache surfaced a broader pattern: in-process module-level mutable state survives test boundaries even when `BIBLIOGON_TEST=1` + `BIBLIOGON_DATA_DIR` correctly isolate the filesystem and DB. Fix: bidirectional `cache_clear()` in every fixture that touches the cached function. An audit script (`TEST-ISOLATION-MODULE-STATE-01`) was added to flag future `@lru_cache` / module-globals introductions without a paired teardown.

**Mutation-testing scope expansion.** The `app/import_plugins/` audit scope grew to cover `overrides.py` bool-coercion survivors, office + wbt walker paths, and the services scope (with a paired test-isolation fix for the regression that surfaced).

**GitHub Actions Node-24 migration.** Every workflow's standard actions bumped to their Node-24-native majors: `actions/checkout` v5, `actions/setup-python` v6, `actions/setup-node` v5, `actions/upload-artifact` v6, `actions/cache` v5, `softprops/action-gh-release` v3, GitHub Pages action trio v5, `actions/configure-pages` v6. The `FORCE_JAVASCRIPT_ACTIONS_TO_NODE24` env-var stays as a safety net. A new lessons-learned section documents the bump methodology (read `action.yml`'s `runs.using:` declaration, NOT release-note prose â€” composite actions need transitivity checks).

**5 new lessons-learned sections** were promoted from this cycle's incidents: multi-tool collaboration tracking (re-sync before accepting new orders), workbox-as-bug-indicator misdiagnosis, user-perceived-bug-vs-code-bug discrimination, in-memory cache test-isolation, and external GitHub Action major-version drift. Plus a sixth ("testid namespace pinning prevents silent E2E skips") promoted from the positive discipline applied to the new article-to-book wizard.
