# Bibliogon v0.34.0 — 2026-05-17

The "Picture-Book Phase 4 foundation + Categories/BISAC + Authors-Database completion + Medium-import v2 preview" release.

Four large feature streams land together: (1) the Picture-Book authoring foundation — `book_type` discriminator on Book, pages CRUD, three-pane PageEditor (thumbnails sidebar + canvas + properties pane), 5 PageLayout variants with distinct visual rendering, per-layout `layout_config` for speech-bubble anchor/opacity/size + image-row positioning/fit/split-ratio, an initial WeasyPrint-based PDF generator skeleton (full route wiring + UI lands in v0.35.0); (2) the Books-only Categories + BISAC subject metadata (KDP-aligned, 9-char BISAC code validation); (3) the completion of the Authors-Database (Phase 1 standalone table + admin tab from the pre-drafted 0.33.1 work, plus Phase 2 Wizard-datalist integration on the Article-to-Book conversion flow); (4) the Medium-import v2 preview-table (uncheck-to-skip + select-all + per-post badges + Playwright smoke). Plus the Comments trash-lifecycle finally arrives in the comments-admin surface (closes the v0.32.0 omission where moved-to-trash comments became invisible), seven user-reported UX bugs (Bug 1, 3, 4a–c, 6, 7, 11), and three new lessons-learned rules.

This release supersedes the pre-drafted v0.33.1 (Comments-Trash recovery + Authors-Database Phase 1) which was never tagged — its content is fully absorbed below.

## Heads-up for users who pressed "Move to Trash" on comments before v0.34.0

**English.** If you moved comments to trash on v0.32.0 or v0.33.0 and couldn't find them afterwards, your data was not lost — the row stayed in the database with a "soft-delete" marker, but the Comments-Admin tab in Settings had no surface to show you the trash or restore from it. v0.34.0 adds that surface. After upgrading, open **Settings → Kommentare** (Comments) and click the new **Papierkorb** (Trash) toggle next to the active list — every previously-trashed comment will appear there, with a per-row **Wiederherstellen** (Restore) button and a bulk-restore affordance.

**Deutsch.** Falls Sie in v0.32.0 oder v0.33.0 Kommentare in den Papierkorb verschoben haben und sie danach nicht mehr finden konnten — Ihre Daten sind nicht verloren. Die Zeile blieb mit einer "Soft-Delete"-Markierung in der Datenbank, aber der Comments-Admin-Tab in den Einstellungen hatte keine Oberfläche, um den Papierkorb anzuzeigen oder wiederherzustellen. v0.34.0 ergänzt diese Oberfläche. Nach dem Upgrade öffnen Sie **Einstellungen → Kommentare** und klicken Sie auf den neuen **Papierkorb**-Schalter neben der aktiven Liste — alle zuvor in den Papierkorb verschobenen Kommentare erscheinen dort, mit einem **Wiederherstellen**-Button pro Zeile und einer Bulk-Wiederherstellung.

## Action required

None. v0.34.0 ships additive surface; every v0.33.0 flow is unchanged.

Four new Alembic migrations land in this release:
- `authors` table (Authors-Database).
- `book_type` discriminator column on `books` + new `pages` table (Picture-Book foundation).
- `books.categories` + `books.bisac_codes` columns (Books-only subject metadata).
- `pages.speech_bubble_config` → `pages.layout_config` column rename (per-layout config generalisation).

All four are forward-compatible. No data migration on existing rows: every legacy book lands as `book_type="prose"` and inherits the existing chapter-based pipeline; `categories` / `bisac_codes` start NULL; Authors-DB starts empty (decoupled from existing free-text `Book.author` / `Article.author`).

## What's new

### Picture-Book Phase 4 foundation (PB-PHASE4 Sessions 2–6 Commit 1)

The picture-book authoring track lands as a usable foundation. Authors can create a picture-book today, lay out pages with five distinct layouts, configure speech-bubble anchor + opacity + size or image-row positioning, and persist everything per-page. The PDF export route lands in v0.35.0 — the WeasyPrint generator skeleton is in place but not yet dispatched from `/api/books/{id}/export/pdf`.

**Schema + API:**
- `book_type` discriminator on `Book` (Literal `"prose"` | `"picture_book"`). Immutable after create — `PATCH /api/books/{id}` returns 400 on mutation attempts.
- New `pages` table + `/api/books/{id}/pages` CRUD (list/create/update/delete/reorder). Reorder is a single transactional batch update.
- Per-page `layout_config` (JSON column) renamed from the original `speech_bubble_config` to generalise across all five layouts.

**Editor:**
- Dashboard split-button: primary "Create Book" + chevron disclosure → "Create Picture-Book" affordance.
- BookEditor `book_type`-based routing — picture-books open into the three-pane `PageEditor` instead of the chapter-based prose editor.
- Three-pane PageEditor: thumbnails sidebar (with @dnd-kit drag-reorder + add-page CTA) + canvas main + properties pane.
- LayoutPicker with primary defaults (`speech_bubble` + `image_top_text_bottom`) and a "More layouts" disclosure for the remaining three.
- 5 PageLayout variants, each with distinct CSS-Module rendering:
  - `speech_bubble` — full-bleed image with a positioned bubble (5 anchor presets: bottom-center default, top-left, top-right, bottom-left, bottom-right; opacity slider 0.3–1.0; size slider 30–80% width).
  - `image_top_text_bottom` — 70/30 vertical split (re-proportioned from initial 60/40 per user feedback).
  - `image_left_text_right` — 60/40 horizontal split.
  - `image_full_text_overlay` — full-bleed image with a positioned text band (text position 3-row preset; text-backdrop opacity slider; image defaults to `object-fit: cover` to prevent letterboxing past the overlay band).
  - `text_only` — no image region.
- LayoutConfig dispatcher in the properties pane — surfaces the correct configuration body per active layout. Debounced (300 ms) for continuous controls (sliders); discrete controls (radios, dropdowns) write immediately.
- Visual containers — text + image regions render with per-layout borders + background tints for spatial clarity in the stacked/split layouts; overlay + speech-bubble layouts skip the borders (their text region has its own backdrop).
- On-image hover overlay for the image-replace button (Notion-style top-right toolbar).

**Metadata integration (Session 5):**
- `BookMetadataEditor` is `book_type`-aware. Chapter-less book types (picture-book) hide the Chapter-templates surface in the General tab.
- PageEditor → Metadata entry-point preserves the URL route; Metadata-button in the page-editor header opens the metadata editor in place of the page-editor pane.

**PDF export skeleton (Session 6 Commit 1):**
- WeasyPrint `^66.0` added to the export plugin's deps (the `^61.0` series silently dropped CSS Grid; `^66.0` re-adds it).
- New `bibliogon_export.picture_book_pdf` module with a `generate_picture_book_pdf(book_data, pages, assets, upload_dir, output_path)` entrypoint. Mirrors PageCanvas's per-layout inline-style logic.
- Route dispatch + UI button + i18n + E2E ship in v0.35.0.

**i18n:** `ui.page_editor.*` namespace across all 8 catalogs (DE + EN native; ES / FR / EL / PT / TR / JA passthru-EN per the `REVIEW_STATUS.md` policy).

**E2E coverage:** picture-book page-editor flow (`e2e/smoke/picture-book-page-editor.spec.ts`), layout-render verification (`e2e/smoke/picture-book-layouts.spec.ts`), per-layout config flows (`e2e/smoke/layout-configuration.spec.ts`), PageEditor → Metadata navigation (`e2e/smoke/picture-book-metadata.spec.ts`).

### Books-only Categories + BISAC subject metadata (Bug 9)

- Schema: `Book.categories` (free-text JSON list, KDP-aligned) + `Book.bisac_codes` (BISAC 9-char codes, regex `^[A-Z]{3}[0-9]{6}$`).
- Marketing-tab wires Categories + BISAC with shared `CategoryInput` + `BisacCodeInput` components.
- KDP plugin metadata-checker extension: format validation + missing-category warning.
- Documented intentional asymmetry: Articles use `Article.topic` (single enum, settings-managed) + `Article.tags` (free-text); Books use `Book.categories` + `Book.bisac_codes`. The two domains have different metadata shapes by design (articles ship to N platforms each with its own tagging norms; books target retail catalogues with industry-standard subject hierarchies).
- BISAC autocomplete + bundled catalog is deferred (`BISAC-DATABASE-LOOKUP-01`, P5) — the licensing cost from BISG ($590/year) is incompatible with the local-first donation-funded model.

### Categories + BISAC Marketing-tab leak hotfix

A regression caught during release prep: the Marketing tab's `forceMount` prop caused the new Categories + BISAC panel to render on every tab, not just Marketing. Cause: `forceMount` actively prevents Radix from applying the `hidden` attribute on non-active tabs. Fix: removed `forceMount` on the Marketing `Tabs.Content`. Vitests switched from `fireEvent.click` to `fireEvent.mouseDown` (Radix `Tabs.Trigger` listens to `onMouseDown` internally, not `onClick`). New tab-content-isolation Playwright spec pins the regression.

### Authors-Database (Bug 8 Phase 1 + Phase 2)

**Phase 1 — storage layer + admin tab** (originally drafted for v0.33.1):
- New standalone `authors` table + `/api/authors` CRUD + a Settings "Autoren-Datenbank" / "Authors Database" tab sibling to the existing personal-identity "Autor" tab.
- Schema: `Author(id, name, slug, bio, created_at, updated_at)`. Slug is server-generated from name (lowercase + hyphenated, German + Nordic diacritics transliterated to ASCII before NFKD-fold for other diacritics, all-emoji fallback to `"author"`, collisions appended with `-2 / -3 / ...`).
- Slug is immutable after create — protects any future `/authors/{slug}` URL routing from breaking on rename.
- DELETE is idempotent (204 on already-gone) — supports multi-tab races.
- Decoupled from existing `Book.author` / `Article.author` columns by design (no FK, no backfill — they remain free-text; the Authors-DB is an opt-in suggestion layer).
- Coverage: 52 new backend pytest cases, 7 `client.test.ts` cases, 17 component Vitest cases, 16 i18n keys across all 8 catalogs.

**Phase 2 — Wizard datalist integration:**
- Article-to-Book conversion wizard Step-1 author field now offers a `<datalist>` populated from the Authors-DB (filtered + sorted by `computeAuthorSuggestions` utility — Vitest-covered).
- Multi-article selections pre-fill the wizard's author field when all selected articles share the same author.
- An "Add to Authors database" checkbox on wizard submit creates a new Authors-DB row from the wizard's author field.
- Playwright smoke spec: `e2e/smoke/wizard-author-dropdown.spec.ts`.
- Documented intentional asymmetry: the Wizard datalist lands Books-only at the conversion surface in v0.34.0. ArticleEditor + BookEditor free-text author inputs stay plain text — the wizard is the high-leverage surface (multi-article selection surfaces multiple authors at once); single-record editors get the datalist in a follow-up release (filed as `AUTHOR-DATALIST-EXTEND-EDITORS-01` P3).

### Medium-import v2 preview-table (Phases 1–4)

A larger Medium-import workflow change: instead of dry-running an entire archive and producing a single result-table at the end, the user uploads → previews a checkbox table of every detected post → unchecks the ones they want to skip → imports the selection. Reduces wasted import cycles when a user wants to skip parts of an archive.

- Backend: preview + selection endpoints for the v2 dry-run workflow (parsing happens once; the same parsed state is consumed by the selection endpoint).
- API client: `mediumImport.preview / importSelected / cancelPreview` helpers.
- Frontend UI: preview-table component + v2 state machine; per-post badges (Article / Comment / Duplicate / Warnings).
- 8-language i18n catalogs + Playwright smoke (`e2e/smoke/medium-import-preview.spec.ts`).

### Comments trash-lifecycle (Bug 10)

Comments shipped soft-delete in v0.32.0 but the counterparts (list-trashed, restore, permanent-delete-from-trash, empty-trash, bulk-restore) were filed as "v2 ships" in the original commit and never picked up. Production smoke surfaced **61 user-trashed comments stuck in invisible purgatory**: the user had pressed "Move to Trash", got a "moved to trash" toast, and then had no surface to look in. v0.34.0 closes the gap with:

- Backend: four new endpoints (`GET /api/comments/trash/list`, `POST /api/comments/trash/{id}/restore`, `DELETE /api/comments/trash/{id}`, `DELETE /api/comments/trash/empty`) + a bulk-restore endpoint (`POST /api/comments/trash/bulk-restore`). Bulk-permanent-delete-from-trash reuses the existing `POST /api/comments/bulk-delete?permanent=true`.
- Frontend: a `viewMode` toggle on Settings → Comments with `comments-active-toggle` and `comments-trash-toggle` buttons. The Trash toggle carries a badge with the trash population count. Per-row actions in trash mode are Restore + Permanent-Delete (replacing the active-view Trash icon). An "Empty trash" CTA in the toggle row + a danger-variant confirm. Bulk affordances (Restore + Permanent-Delete) in trash view via a dedicated `comments-trash-bulk-action-bar`.
- Coverage: 22 new backend pytest cases, 22 new frontend Vitest cases, 4 Playwright smoke specs (`e2e/smoke/comments-trash-lifecycle.spec.ts`).

### UX bug fixes from manual smoke testing

- **Bug 1**: Settings/Help/GetStarted back-button uses browser history (not hardcoded `/`) — Vitest + Playwright regression pins.
- **Bug 3**: independent view-mode defaults for AD-Trash + BD-Trash — previously both inherited the same default, ignoring per-tab persistence.
- **Bug 4a–c**: Comments-Admin bulk-delete affordance + `CommentPreviewModal` with click-row-to-open + Reclassify-in-preview-only refactor + 8-language i18n + Playwright smoke. Closes the Articles-vs-Books-vs-Comments parallel-surface asymmetry tally to 7 items now confirmed.
- **Bug 6**: menu items must not `preventDefault` on dialog-trigger — `onSelect` handlers that open a dialog must let the menu close first, otherwise dialog floats above a still-visible menu (violates "one modal surface at a time" UX contract). Fixed across 6 callsites; Playwright regression pin.
- **Bug 7**: mock-contract repair for `ArticleList` trash view-mode test — new hook + new mock-key contract drift; new lessons-learned rule.
- **Bug 11**: BookDashboard list-view selection checkboxes (parity with Articles list-view; closes Articles-vs-Books-asymmetry occurrence #6).

### Picture-Book bug fixes from mid-release smoke (Session 4c-A)

- **Bug B** (image_full_text_overlay image letterboxes past text container bounds): image defaults to `object-fit: cover` so the absolutely-positioned text band (left:0 right:0) aligns to the visible image edges. A future `image_fit` knob in `LayoutConfigImageRow` can override this per-page.
- **Bug A + Bug C** (cross-layout key accumulation): `layout_config` is a single flat JSON dict that accumulated keys from every layout the page had ever worn. Switching `speech_bubble → image_full_text_overlay` left the bubble's `image_fit: "cover"` co-resident with the new layout's keys; `PageCanvas`'s per-layout style resolver picked up the stale key and cropped the photo unexpectedly. Inspecting one production page revealed 8 keys mixed from three different layouts. **Fix A** (this release): purge `layout_config` to `null` in the same `api.pages.update` payload that flips the layout. Atomic; no intermediate state. Trade-off: user loses per-layout config on switch. **Fix B** (deferred to 4c-B): namespace `layout_config` per layout so each layout's config survives a switch + return.

### Lessons-learned (3 new rules)

1. **"Half-wired feature lifecycle"** extended to cover the frontend shape — state-write without state-consumer (renderer/reader) is purgatory just like backend soft-delete without restore. Detection grep: `grep -rnE 'out of scope|v2 ships|deferred to v2' backend/app/ frontend/src/ plugins/`.
2. **"Half-wired trash lifecycle"** as canonical concrete example (Bug 10): soft-delete shipped without the restore-surface is data purgatory, not a feature. Deferred-half work MUST be filed as a load-bearing backlog item with a real ID, not as docstring prose.
3. **"Test-isolation discipline"**: never run integration smoke-tests outside pytest. The harness's three-layer isolation (`BIBLIOGON_TEST=1` env-var, `:memory:` DB URL, `.bibliogon-production` marker tripwire) only fires under pytest. A bare `poetry run python -c "from app.main import app"` bypasses every guard and points at the real production DB.
