It’s been seventeen days since the v0.19.1 article. Bibliogon shipped v0.20.0 through v0.30.0 in that window: eleven feature releases plus a seven-tag hotfix cluster that taught us something worth carrying forward.

Bibliogon is a book authoring tool that keeps your manuscript on your own disk. You install it via Docker or a native launcher, you write in a browser-based TipTap editor, you export to KDP-ready EPUB and PDF, and you own every file the app produces. Article authoring landed in this period, the launcher now speaks eight languages, and there’s now a Git-based backup system that gives you Git’s branching and history model on top of the local-first storage.

This piece is the follow-up to the v0.19.1 article and assumes you read the first one. If you didn’t, the headline is: open-source, MIT, your manuscript stays on your disk, your AI provider is your choice, your plugin escape hatch is real Python.

A production data loss, and what we did about it

The most important thing that happened in this period wasn’t a feature ship. In April, a test path made it into production fixtures, and the test setup wiped a real database. Books gone, chapters gone, assets gone. Recovered from a backup, but only after a bad twenty minutes and a longer post-mortem.

What shipped in response, the part that matters now, is two tripwires that make the same incident impossible going forward.

A .bibliogon-production marker file lives in the production data directory. The FastAPI lifespan writes it on first start in non-test mode. If any test ever sees this marker (backend/tests/conftest.py checks for it before any app.* import), the entire test run aborts with pytest.exit(returncode=2). Exit code 2 means: a test path was pointed at real data; investigate, do not delete the marker to make the test pass.

A separate test-mode env-var (BIBLIOGON_TEST=1) is set before any app.* import, redirects all path resolution into a tmp_path_factory-managed dir, and forces the database to sqlite:///:memory: regardless of what other env vars are set.

Both tripwires are documented in .claude/rules/lessons-learned.md and in the architecture page on the public docs site, with the dated incident reference. Honest about the close call, concrete about the fix.

The same principle then surfaced again at smaller scale: the v0.26 release tag failed CI on a chmod bit, then on a PyInstaller spec error, then on a mypy escape, then on a ruff-format nit. Four mechanical hotfixes (v0.26.1, .2, .3, .4) before v0.26.5 stuck. The lesson, that every “this is enforced” claim deserves an adversarial check that confirms enforcement actually works, turned into a mandatory pre-tag verify chain (make test, tsc, vitest, playwright smoke, ruff, mypy, pre-commit, pyinstaller spec --clean) that has held across every release since v0.27.

What’s new since v0.19.1

Article authoring (Phase 2, v0.18 to v0.22). Bibliogon now distinguishes Articles from Books as separate entities. Articles get a dedicated TipTap editor without the chapter sidebar, multi-platform publication tracking (Medium, Substack, X, LinkedIn, dev.to, Mastodon, Bluesky, custom), drift detection that flags non-synchronized publications when the article content changed after publishing, and a “Live bestätigen” action to reset the comparison. Promo-posts ship as Publications with an is_promo flag. The whole shape is a separate Article model alongside Book rather than overloading Book; the architecture decision (formerly AR-02) resolved as Option B.

Bulk export (v0.27 articles, v0.28 books). The Articles dashboard exports multiple articles in one operation: per-tile checkboxes, Select-all, sticky bulk-action bar, format dropdown (Markdown, HTML, PDF, DOCX), output mode (ZIP-of-individuals or combined document). 200-item hard limit. The Books dashboard reaches feature parity in v0.28 with three formats (EPUB, PDF, DOCX), ZIP-only by design. Combining N books into one EPUB would have to decide whose metadata wins and which book contributes the cover, which is not a natural author workflow.

Git-based backup (v0.21.0). Five-phase plan, all shipped: local Git per book, remote push/pull with encrypted PATs, SSH key generation in-app, three-way merge with per-chapter conflict resolution UI, Markdown side-files for diff-friendly storage. Authors who want Git’s branching and history model on their manuscripts now have it. The plugin-git-sync package layers a different ownership model on top for users who want their books mirrored to a separate repo without going through core Git integration.

AI Review extension (v0.20.0). Async chapter review against any of the five providers (Anthropic, OpenAI, Google, Mistral, LM Studio). Reports are persistent Markdown files in the project structure. Prompts are chapter-type-aware: a Preface gets reviewed differently than a Chapter than an Acknowledgments section.

Launcher: 8 languages, first-run UX overhaul, cross-platform install scripts. The launcher binary went from English-only to bilingual (en/de) to fully eight-language (en/de/el/es/fr/pt/tr/ja) over four releases. OS-locale auto-detection picks the right language; de_DE/de_AT/de_CH all resolve to de, pt_BR and pt_PT to pt, etc. Three of the six new catalogs (Greek, French, Spanish) are user-validated; the other three (Portuguese, Turkish, Japanese) ship with explicit _meta.review_status: "pending native speaker" markers and a public call-for-reviewers GitHub issue. Marker enforcement is machine-checked: parity tests fire on every push asserting that pending-review catalogs carry the marker and validated ones don't.

Alongside the localization, the launcher gained a welcome dialog for first-run users, a Docker-missing flow with platform-specific install pointers, and three new entry points besides install.sh: install.command (macOS Finder double-click), install.ps1 (PowerShell mirror), install.cmd (Windows double-click that bypasses Group-Policy-locked ExecutionPolicy). All four ship unsigned per launch decision; Gatekeeper and SmartScreen warnings are documented inline.

Data directory swap to platformdirs (v0.25.0). Production data moved out of the project tree to ~/.local/share/bibliogon/ (Linux/macOS) or %LOCALAPPDATA%\bibliogon\ (Windows). Auto-migration on first start, with .migrated-YYYY-MM-DD breadcrumbs at each old path so users can verify the move before deleting. Users with BIBLIOGON_DATA_DIR set in env keep their override; users with the legacy BIBLIOGON_DB_PATH set went through a three-release deprecation cycle (warning v0.27, precedence flip v0.28, full removal v0.30).

Frontend toolchain modernization. React 18 to 19, TypeScript 5 to 6, Vite 7 to 8 (with the new Rolldown bundler), @types/node ^22 to ^24 with tsconfig target/lib ES2020 to ES2022. Cleared four high-severity dev-only audit findings (workbox-build, @rollup/plugin-terser, serialize-javascript: GHSA-5c6j-r48x-rmvq RCE plus GHSA-qj8w-gfj5-8c6v DoS) via the vite-plugin-pwa@1.3.0 upgrade. npm audit --audit-level=high reports zero high-or-above vulnerabilities.

What got harder, and stuck

The pre-v0.30.0 retrospective (docs/journal/retrospective-pre-v0.30.0.md) documented a worked example of the "verify the gate before trusting it" principle: mkdocs build --strict does not actually fail on orphan pages because the "not in nav" message is INFO-level, which strict mode ignores. Two pages sat orphan in the public docs site for two release cycles before a manual audit caught them.

The fix series that followed, five gates at four lifecycle layers (commit-time pre-commit hook, PR/CI bilingual parity test, release-time aggregated check, marker-block nav single-source-of-truth eliminating the hand-maintained-twice failure mode), was the kind of thing the original article would have called “the janitor pass.” Same shape repeated for plugin lockfile drift after a CI red-on-main: make test was green because the backend's combined lockfile resolved every plugin via path-deps, but CI installs each plugin against its own lockfile and that lockfile lagged.

The generalization is now a documented lesson: any time there are two installation paths for the same code, both must be tested at gate time. The hooks that enforce it shipped within hours of the discovery.

What this still doesn’t do

Same honest gaps as April:

  • No real-time collaboration. Local-first rules that out.
  • No mobile app. UI adapts to tablet widths.
  • Not Vellum-polished. Exports meet KDP specs and are readable; the premium typesetting territory belongs to dedicated tools.
  • Onboarding is still rough. The launcher and one-liner work, but first-run assumes a user who has seen a terminal before.

How to try v0.30.0

One-liner (Docker required):

curl -fsSL https://raw.githubusercontent.com/astrapi69/bibliogon/main/install.sh | bash

Windows (PowerShell):

irm https://raw.githubusercontent.com/astrapi69/bibliogon/main/install.ps1 | iex

macOS double-click: clone the repo, then double-click install.command in Finder. Windows double-click: same with install.cmd.

Native launcher: download bibliogon-launcher.exe (Windows), bibliogon-launcher-macos.zip (macOS), or bibliogon-launcher-linux (ELF) from the v0.30.0 release page. Each ships with a .sha256 checksum. Unsigned; first-run shows a Gatekeeper or SmartScreen warning.

Docs at https://astrapi69.github.io/bibliogon/ now cover Architecture, Contributing, Deployment, API reference, and the plugin development guide as bilingual core pages, alongside the per-platform install guides.

What’s next

TipTap 2 to 3 migration is blocked on an upstream npm publish (@sereneinserenade/tiptap-search-and-replace@0.2.0); next re-audit 2026-06-02. A prosemirror-search adapter (~50-80 LOC) is on the table as the alternative unblock and has not been authorized.

The pt/tr/ja launcher catalogs need native-speaker review. The public call-for-reviewers is GitHub issue #18; the marker stays until a native pass lands or the 2026–08–07 decision threshold forces an explicit “accept as canonical” call.

Cross-platform installer scripts shipped without fresh-machine validation. The macOS Apple-silicon and Windows 11 fresh-account runs are queued; first user report folds into the next point release.

Join in

The MIT license, the issue tracker, the documented hook specs, and the eight-language UI are all the same shape they were in April. The surface just grew.

If you write in Portuguese, Turkish, or Japanese and you can spend an hour reviewing a 95-key catalog, that’s the most concretely useful contribution available right now (issue #18 is the entry point). If you publish through KDP and find a validation case the plugin misses, the issue tracker is open. If you want to read how the plugin system works, docs/help/en/developers/plugins.md is the 822-line guide.
