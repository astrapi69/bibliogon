## Archived 2026-05-07 (DEP-DBPATH-01 step 3)

### DEP-DBPATH-01 step 3 — BIBLIOGON_DB_PATH override removed

Closes the `BIBLIOGON_DB_PATH` deprecation cycle begun in v0.27.0.
Step 1 (deprecation warning, v0.27.0) and step 2 (precedence flip
making `BIBLIOGON_DATA_DIR` the senior source, v0.28.0) shipped in
prior releases. Step 3 — full removal of the override — lands
between the v0.29.0 tag and the next release tag, accumulated on
`main`.

**Behavior change.** `BIBLIOGON_DB_PATH` is no longer honoured for
path resolution. The resolver in `backend/app/database.py`
short-circuits the env var: if it is set in the environment, a
single warning is logged at startup naming the ignored value, and
path resolution proceeds through the normal `BIBLIOGON_DATA_DIR`
or platformdirs default chain. The warning will be removed in a
later release once existing deployments have migrated.

The new resolution priority is:

1. `BIBLIOGON_TEST=1` → `TEST_DATABASE_URL` or `:memory:`.
2. `DATABASE_URL` (verbatim).
3. `BIBLIOGON_DATA_DIR` derivation → `<DATA_DIR>/bibliogon.db`.
4. Platformdirs default via `app.paths.get_db_path()` (Linux/macOS
   `~/.local/share/bibliogon/`, Windows `%LOCALAPPDATA%\bibliogon\`,
   Docker `/app/data` via the container env-var).

**Tests.** `backend/tests/test_database_url_resolution.py`
rewritten to match the new behavior (6 tests, all green):

- `test_db_path_alone_is_ignored_with_warning` — DB_PATH alone:
  resolver falls through to platformdirs default; warning emitted.
- `test_db_path_with_data_dir_is_ignored_with_warning` — both
  set: DATA_DIR-derived path used; same warning emitted.
- `test_data_dir_alone_does_not_warn` — clean DATA_DIR path; no
  warning.
- `test_neither_env_var_set_does_not_warn` — bare default path;
  no warning.
- `test_database_url_takes_precedence_over_db_path` — DATABASE_URL
  short-circuits including the DB_PATH ignore-warning path.
- `test_test_mode_short_circuits_all_overrides` — BIBLIOGON_TEST
  forces `:memory:` regardless of the other vars.

**Docs.** README.md, README-de.md, and the bilingual deployment
guide (`docs/help/{en,de}/developers/deployment.md`) updated to
mark `BIBLIOGON_DB_PATH` as **Removed in v0.30.0** with the full
deprecation timeline (warning v0.27.0, precedence flip v0.28.0,
removal v0.30.0). The variable row stays in the env-var tables
for one release cycle so users with existing `BIBLIOGON_DB_PATH=`
in their environment can find an explicit "this no longer works"
entry rather than wondering why the doc forgot about it. A later
release drops the row entirely.

**No Docker / launcher / installer impact.** `docker-compose.prod.yml`
sets only `BIBLIOGON_DATA_DIR=/app/data`; no installer script
references DB_PATH. Existing default deployments are unaffected;
only deployments that historically set `BIBLIOGON_DB_PATH` need to
remove it from their environment (the warning at startup names
the value).

**Verification.** `make test` green (backend 1298 + 1 skipped),
ruff + mypy clean, DE content sweep clean (no umlaut
transliterations introduced in the new doc rows).

The next release tag will pick up step 3 along with the post-
v0.29.0 backlog items (docs+i18n session and EL/ES/FR i18n
corrections) and ship as v0.30.0.

### DEP-09 — Vite 7 -> 8 upgrade

Frontend bumped to Vite 8.0.10 + vite-plugin-pwa 1.3.0 + vitest
4.1.5 + @vitest/coverage-v8 4.1.5. Unblocked the same day by
upstream `vite-plugin-pwa@1.3.0` adding Vite 8 to its peer-dep
range (`^3.1.0 || ^4 || ^5 || ^6 || ^7 || ^8`). One Vite 8
breaking change required a code edit: Rolldown (Vite 8's
default bundler) only accepts the function form of
`manualChunks`, not the legacy object form Vite 7's Rollup
supported. `vite.config.ts` `manualChunks` was refactored from
a `{chunkName: [pkgs]}` literal to a function that matches the
module id against a packages-to-chunk map and returns the
bucket name; trailing-slash matching prevents prefix collisions
between sibling packages (`react` vs `react-dom` vs
`react-router-dom`). Build green: 4 chunks emitted with the
same shape Vite 7 produced (vendor-react ~190 KB,
vendor-tiptap ~401 KB, vendor-ui ~218 KB, index ~968 KB),
PWA SW regenerates with 37 precache entries. Test deltas:
frontend 707 passed unchanged; smoke Playwright 191 passed
unchanged. lessons-learned.md updated with the manualChunks
breaking-change note and the stale-dev-server pollution lesson
caught during this session. Will land in the v0.29.0
CHANGELOG.

### D-05: Auto-install Docker Desktop — closed as won't-fix

Original ROADMAP P4 entry rescoped on 2026-05-04 (commit
`93af0b2`) from "full Windows installer (Docker Desktop + repo +
`.env`)" to "Docker Desktop auto-install only", because the
Bibliogon launcher already handles repo download, `.env`
generation, Docker image build, and browser open on first run
via `_install_or_welcome` / `_run_install_flow` in
`launcher/bibliogon_launcher/__main__.py`. Only Docker Desktop
itself remained terminal-required for fresh-machine setups.

Closed as **won't fix** on 2026-05-05 after the installer
discovery session
(`docs/explorations/installer-discovery-report.md`) confirmed
that the Docker Subscription Service Agreement Section 2.1's
non-sublicensable license grant forbids third-party silent
install of Docker Desktop. Solo-OSS-dev strict-EULA-reading
default applies; no legal-review budget exists to chase an
ambiguous interpretation.

The launcher's existing **detect-and-instruct** flow (open the
official Docker download page when Docker is missing, continue
the install once Docker is present) is the EULA-compliant
ceiling. Replacing it with a third-party silent install would
violate Docker's terms.

**Citation:** see
[docs/explorations/installer-discovery-report.md](../explorations/installer-discovery-report.md)
section "Docker Desktop integration" for the cited EULA
passages and the analysis chain.

**Related but NOT subsumed:** `LAUNCHER-SELFREPLACE-01` remains
an open P4 backlog item. Discovery confirmed it is a separate
scope (atomic launcher binary self-replace on Windows is
unrelated to Docker Desktop installation).
