## Archived 2026-05-11 (MEDIUM-IMPORT-CONTENT-TRUNCATION-01 + MEDIUM-IMPORT-LANGUAGE-DETECTION-01)

### MEDIUM-IMPORT-CONTENT-TRUNCATION-01 — walker section-inner truncation

User reported "einige" (a few) Medium articles imported with
truncated content; pointed at "Wie Deutschland seine Unternehmer
ausbeutet" as the canonical case. Systematic survey of the
209-post production import revealed the actual scope was 117
articles (56%) with content loss, including 9 imports with zero
text. Root cause: `walker._walk_body` called
`section.find("div", class_="section-inner")` which returned
only the FIRST inner div per `<section class="section--body">`.
Medium's standard header-image layout puts THREE inner divs
per section (title, image lane, main body); the walker's
title-skip then dropped the only inner it had picked up, and
the main body was never processed.

Fix in commit `ce9665e`: iterate `find_all` over all inner divs.
Verified against the production worst case: pre-fix 170/3567
words (5%); post-fix 3562/3567 (99.9%).

Retro-fix in commit `18c2031`: re-walk source HTML for affected
articles. Materiality gate skips articles where the new content
is less than 2x the existing AND adds fewer than 100 words.
Image URL preservation: matches recovered ImageRefs against
existing ArticleAsset rows so already-downloaded images keep
their local URLs. Applied to the production corpus: 73 articles
materially recovered; 136 already complete; idempotent re-run
shows zero changes.

### MEDIUM-IMPORT-LANGUAGE-DETECTION-01 — auto-detect language via langdetect

User reported "some German articles correctly detected as
German" — DB inspection revealed the walker did NO language
detection and the importer hardcoded "en". The 2 German rows in
the production import had `updated_at > created_at`: user had
manually corrected them in the editor.

Fix in commit `bdf5c14`: walker runs langdetect over the body
text, surfaces `ParsedPost.detected_language`. Importer reads it
with `default_language` fallback. Confidence floor at 0.85
(picked from the production corpus where confident German /
English / Greek detections all scored ~0.99). Body text only,
not title/subtitle. Empty or <50-char bodies return None.

Retro-fix in commit `3af8ecb`: applies langdetect to existing
rows where `language == "en"` (the historical default).
Manually-corrected rows are preserved. Applied to the
production corpus: 81 rows updated (80 German + 1 Greek), 2
manual corrections preserved, 126 confirmed English. Zero
low-confidence rejections.

Final language distribution after retro-fix: 126 en + 82 de + 1 el.

## Archived 2026-05-07 (PLUGIN-LOCKFILE-DRIFT-01)

### PLUGIN-LOCKFILE-DRIFT-01 — gate the two-installation-paths divergence

Closes the discipline gap surfaced earlier in the same day: the
v0.30.0 release CI red-on-main from plugin lockfile drift after
the pre-v0.30.0 fastapi 0.135 → 0.136 dep sweep. Same shape as
MKDOCS-DISCIPLINE-01 (verify-the-gate); separate item because
the affected installation paths differ.

The shape under discipline: `make test` installs plugins via the
backend's combined `poetry.lock` (path-deps), but CI runs
`poetry install --no-interaction --no-ansi` **per plugin** against
THAT plugin's own `poetry.lock`. When a shared external pin
(e.g. fastapi) bumps in every plugin's pyproject but only the
backend lock gets regenerated, `make test` stays green while CI
turns red. The hotfix `3232fad` re-locked all 10 plugin lockfiles
manually.

Four commits address the gap at three lifecycle layers (commit,
gate-time, post-incident verify):

1. **`1b43aec` feat(make): lock-all-plugins + verify-plugin-locks.**
   Two new Makefile targets (for-loop over `plugins/bibliogon-
   plugin-*/`, per pre-inspection Q1: uniform operation, no
   per-plugin variation that would justify the existing
   per-plugin shortcut convention).
   - `lock-all-plugins`: iterates every plugin and runs
     `poetry lock`. Use after any shared-dep pin bump.
   - `verify-plugin-locks`: runs `poetry install --dry-run
     --no-interaction --no-ansi` per plugin, greps for
     "changed significantly", exits 1 with remediation hint.
     NOT in pre-tag chain (per Q4) — manual diagnostic only,
     since the pre-commit hook + CI cover the right times.
2. **`8f6fcea` feat(pre-commit): plugin pyproject.toml must be
   staged with its poetry.lock.** New local hook
   `plugin-lock-paired-with-pyproject` scoped via
   `files: ^plugins/bibliogon-plugin-[^/]+/pyproject\.toml$`.
   `scripts/check_plugin_lock_paired.py` consults
   `git diff --cached --name-only` to enumerate the staged
   set, asserts each hook-supplied pyproject path's sibling
   `poetry.lock` is also staged. Pyprojects without paired
   staged lockfile fail with a remediation hint covering the
   v0.30.0 incident, the `make lock-all-plugins` workflow, and
   a pointer to the lessons-learned entry. Per Q2: only
   pyproject-staging triggers; lockfile-only commits (e.g.
   follow-up after `poetry lock`) are valid and do not fire.
3. **`e31c4fd` test(pre-commit): hook self-check.** 6 tests in
   `backend/tests/test_plugin_lock_drift_hook.py` exercising
   the hook against an isolated git work-tree per test
   (option 3a per Q3 — the cheap path; the heavy path stays as
   the existing CI plugin matrix). Each test stages files,
   runs the hook, asserts exit code + key strings in stderr.
   All 6 tests pass in 0.22s. Joins backend pytest, so the
   gate fires on every `make test` (local + CI).
4. **`<this commit>` chore(backlog): close PLUGIN-LOCKFILE-DRIFT-01.**

The three gates fire at three lifecycle layers:

- **Commit time:** the pre-commit hook (commit 2) catches a
  plugin pyproject staged without its lockfile the moment a
  contributor stages it.
- **PR / CI time:** the existing `poetry install --no-interaction`
  per-plugin matrix in CI (unchanged by this work) is the heavy
  drift detection that catches actual lockfile content drift.
- **Manual diagnostic:** `make verify-plugin-locks` for power
  users who want to verify before push without invoking the
  full `make test`.

Lessons-learned cross-link: the morning's "Two installation
paths diverge: `make test` vs per-plugin CI" entry already
captures the meta-rule. This commit adds a one-line
"now enforced via PLUGIN-LOCKFILE-DRIFT-01" pointer to that
entry so future readers find the enforcement path from the
diagnosis.

Same-day archive (per Q6 / continuous-archival convention):
this entry archives the work in the same commit that closes
the backlog item. The PLUGIN-LOCKFILE-DRIFT-01 entry was
filed in commit `141b0f6` ~30 minutes ago and is closed here.

**Verification chain:**
- `make verify-plugin-locks` on current state: OK (10 plugins
  in sync after the morning's hotfix).
- `make test`: backend 1302 → 1307 (+5; 6 new test functions,
  one of them shares an assertion path so net +5), 1 skipped,

### CI hotfix: plugin lockfile drift after fastapi 0.135 -> 0.136

User flagged CI + Coverage red on main from the v0.30.0 release commit
`be4b6f3`. Root cause: the pre-v0.30.0 dep sweep bumped
fastapi `^0.135.0 → ^0.136.0` in 11 pyproject.toml files but only ran

Hotfix: `3232fad fix(ci): regenerate plugin lockfiles after fastapi
0.135 -> 0.136 bump`. Re-ran `poetry lock` in each of 10 plugin
directories. Verified green on the next CI run.

The shape — `make test` green but CI red — is the same shape as the
v0.30.0 retro's "verify the gate before trusting it" finding. New
lessons-learned entry added: "Two installation paths diverge:
`make test` vs per-plugin CI". Tracked as `PLUGIN-LOCKFILE-DRIFT-01`
(P3) for a follow-up session: `make lock-all-plugins` target + a CI
gate that flags staged plugin pyproject without paired lockfile

### MKDOCS-DISCIPLINE-01 — five discipline improvements

Closes the docs-toolchain gap surfaced in the pre-v0.30.0
retrospective: `mkdocs build --strict` did not actually fail on
orphan pages because "not included in the 'nav' configuration" is
INFO-level, which strict mode ignores. Two pages
(articles/bulk-export.md, install/docker-desktop.md) sat orphan in
the public docs site for two release cycles before the docs+i18n
drift audit caught them.

Six commits address the gap:

1. **`bd0f554` fix(docs): detect orphan pages in mkdocs build via
   grep.** `scripts/check_mkdocs_orphans.sh` runs `mkdocs build
   --strict`, captures output, greps for the exact string
   `not included in the "nav" configuration`, exits non-zero if
   any orphan is found outside the explicit ALLOWLIST (single
   entry today: `SCREENSHOTS-TODO.md`).
2. **`b65e9c6` feat(docs): generate mkdocs.yml nav from
   _meta.yaml via marker blocks.** `scripts/generate_mkdocs_nav.py`
   replaced the previous whole-file-rewrite implementation with
   marker-block replacement (preserves comments + structure
   outside the marker pairs). Two marker pairs:
   `AUTO-GENERATED-NAV-START/END` inside the `nav:` block (after
   the hand-managed `Home: de/index.md` line), and
   `AUTO-GENERATED-NAV-TRANSLATIONS-START/END` inside the i18n
   plugin's `nav_translations:` block. `--check` mode exits 1 on
   drift with a unified-diff to stderr. New make targets:
   `sync-mkdocs-nav`, `verify-mkdocs-nav`, `check-mkdocs-orphans`,
   `verify-docs-discipline` (aggregate).
3. **`3315137` chore(release): add mkdocs nav verify + orphan
   check to pre-tag chain.** `release-workflow.md` Step 5 (Tests)
   bash chain now includes `make verify-docs-discipline` between
   `pre-commit run --all-files` and the launcher pyinstaller
   smoke build. Final-checklist item also added.
4. **`7e62979` feat(docs): pre-commit hook ensures help pages are
   in _meta.yaml.** `scripts/check_help_meta_completeness.py` runs
   under `.pre-commit-config.yaml` `local` repo, scoped via
   `files: ^docs/help/(en|de)/.*\.md$`. Fails when a staged help
   page's slug is missing from `_meta.yaml` and prints a YAML
   snippet the contributor can paste. Allowlist:
   `index.md` (landing pages have a different contract per
   pre-inspection Q2).
5. **`9ea4ff6` test(docs): enforce bilingual parity for help
   pages.** `backend/tests/test_docs_parity.py` (3 tests) walks
   `docs/help/en/` and `docs/help/de/`, asserts the file sets
   match modulo `EXCEPTIONS_INDEX_BASENAMES = {"index.md"}`. Zero
   asymmetries on v0.30.0 state — test passes out of the gate.
   Joins the backend pytest suite, so the gate fires on every
   `make test` (local + CI), not only at release time.
6. **`<this commit>` chore(backlog): close MKDOCS-DISCIPLINE-01,
   add PLUGIN-LOCKFILE-DRIFT-01, lessons-learned entry.**

The five discipline improvements together address the gap at four
different layers of the lifecycle:

- **Commit time:** the pre-commit hook (commit 4) catches a new
  page added without an `_meta.yaml` entry the moment a
  contributor stages it.
- **Pull-request / CI time:** the bilingual parity test (commit
  5) runs on every push as part of the backend pytest suite.
- **Release time:** the pre-tag chain (commit 3) bundles
  `verify-mkdocs-nav` + `check-mkdocs-orphans` (commits 1+2) as
  a MANDATORY gate before tagging.
- **Documentation drift:** the SoT pattern (commit 2) eliminates
  the hand-maintained-twice failure mode by deriving mkdocs.yml
  from `_meta.yaml` deterministically.

The retro's commitment "verify the gate before trusting it" turns
into a concrete property: `mkdocs build --strict` exits 0 with
orphan pages, so we wrote our own gate (commit 1) that actually
fails the build. A future version of mkdocs-strict mode that
escalates orphan-INFO to WARNING would let us drop our wrapper —
but until then, the wrapper is the gate.

## Archived 2026-05-07 (DEP-DBPATH-01 step 3)

### DEP-DBPATH-01 step 3 — BIBLIOGON_DB_PATH override removed

Closes the `BIBLIOGON_DB_PATH` deprecation cycle begun in v0.27.0.
Step 1 (deprecation warning, v0.27.0) and step 2 (precedence flip
making `BIBLIOGON_DATA_DIR` the senior source, v0.28.0) shipped in
prior releases. Step 3 — full removal of the override — lands
between the v0.29.0 tag and the next release tag, accumulated on
`main`.

**Behavior change.** `BIBLIOGON_DB_PATH` is no longer honoured for
path resolution. The resolver in `backend/app/database.py`
short-circuits the env var: if it is set in the environment, a
single warning is logged at startup naming the ignored value, and
path resolution proceeds through the normal `BIBLIOGON_DATA_DIR`
or platformdirs default chain. The warning will be removed in a
later release once existing deployments have migrated.

The new resolution priority is:

1. `BIBLIOGON_TEST=1` → `TEST_DATABASE_URL` or `:memory:`.
2. `DATABASE_URL` (verbatim).
3. `BIBLIOGON_DATA_DIR` derivation → `<DATA_DIR>/bibliogon.db`.
4. Platformdirs default via `app.paths.get_db_path()` (Linux/macOS
   `~/.local/share/bibliogon/`, Windows `%LOCALAPPDATA%\bibliogon\`,
   Docker `/app/data` via the container env-var).

**Tests.** `backend/tests/test_database_url_resolution.py`
rewritten to match the new behavior (6 tests, all green):

- `test_db_path_alone_is_ignored_with_warning` — DB_PATH alone:
  resolver falls through to platformdirs default; warning emitted.
- `test_db_path_with_data_dir_is_ignored_with_warning` — both
  set: DATA_DIR-derived path used; same warning emitted.
- `test_data_dir_alone_does_not_warn` — clean DATA_DIR path; no
  warning.
- `test_neither_env_var_set_does_not_warn` — bare default path;
  no warning.
- `test_database_url_takes_precedence_over_db_path` — DATABASE_URL
  short-circuits including the DB_PATH ignore-warning path.
- `test_test_mode_short_circuits_all_overrides` — BIBLIOGON_TEST
  forces `:memory:` regardless of the other vars.

**Docs.** README.md, README-de.md, and the bilingual deployment
guide (`docs/help/{en,de}/developers/deployment.md`) updated to
mark `BIBLIOGON_DB_PATH` as **Removed in v0.30.0** with the full
deprecation timeline (warning v0.27.0, precedence flip v0.28.0,
removal v0.30.0). The variable row stays in the env-var tables
for one release cycle so users with existing `BIBLIOGON_DB_PATH=`
in their environment can find an explicit "this no longer works"
entry rather than wondering why the doc forgot about it. A later
release drops the row entirely.

**No Docker / launcher / installer impact.** `docker-compose.prod.yml`
sets only `BIBLIOGON_DATA_DIR=/app/data`; no installer script
references DB_PATH. Existing default deployments are unaffected;
only deployments that historically set `BIBLIOGON_DB_PATH` need to
remove it from their environment (the warning at startup names
the value).

**Verification.** `make test` green (backend 1298 + 1 skipped),
ruff + mypy clean, DE content sweep clean (no umlaut
transliterations introduced in the new doc rows).

v0.29.0 backlog items (docs+i18n session and EL/ES/FR i18n

### D-05: Auto-install Docker Desktop — closed as won't-fix

Original ROADMAP P4 entry rescoped on 2026-05-04 (commit
`93af0b2`) from "full Windows installer (Docker Desktop + repo +
`.env`)" to "Docker Desktop auto-install only", because the
Bibliogon launcher already handles repo download, `.env`
generation, Docker image build, and browser open on first run
via `_install_or_welcome` / `_run_install_flow` in
`launcher/bibliogon_launcher/__main__.py`. Only Docker Desktop
itself remained terminal-required for fresh-machine setups.

Closed as **won't fix** on 2026-05-05 after the installer
discovery session
(`docs/explorations/installer-discovery-report.md`) confirmed
that the Docker Subscription Service Agreement Section 2.1's
non-sublicensable license grant forbids third-party silent
install of Docker Desktop. Solo-OSS-dev strict-EULA-reading
default applies; no legal-review budget exists to chase an
ambiguous interpretation.

The launcher's existing **detect-and-instruct** flow (open the
official Docker download page when Docker is missing, continue
the install once Docker is present) is the EULA-compliant
ceiling. Replacing it with a third-party silent install would
violate Docker's terms.

**Citation:** see
[docs/explorations/installer-discovery-report.md](../explorations/installer-discovery-report.md)
section "Docker Desktop integration" for the cited EULA
passages and the analysis chain.

**Related but NOT subsumed:** `LAUNCHER-SELFREPLACE-01` remains
an open P4 backlog item. Discovery confirmed it is a separate
scope (atomic launcher binary self-replace on Windows is
unrelated to Docker Desktop installation).
