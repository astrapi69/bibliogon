### Bug A close-out: workbox "broken" was a perception-lag UX bug

2026-05-14 user report: "Articles-Trash Restore button is
broken; workbox blocks the request". Investigation chain:

1. SW config audit — single `urlPattern: /^\/api\//` registered
   for GET only. Every non-GET API call triggers
   `No route found for: <url>` as informational pass-through
   (not blocking).
2. Manual smoke — restore POST fires, backend processes
   correctly, DB confirms article restored.
3. Real cause — `[Violation] 'click' handler took 419ms`
   browser violation log. Restore worked correctly; the
   handler chains two network roundtrips (`POST .../restore`
   + `GET /articles`) with synchronous `setTrash` + `setArticles`
   state updates in between. 419ms registers as "slow + no
   clear success signal" because the user stays in the
   trash view, a transient toast fires, and the restored row
   vanishes from the filtered list.

Outcome: no code change, two artefacts.

- **`RESTORE-UX-FEEDBACK-01`** (P3 IMPROVEMENT) filed in
  `docs/backlog.md` for the optimistic-update + clearer
  post-restore feedback redesign. Trigger: second user report
  with the same "feels broken" framing.
- **Lessons-learned promotion**: rule
  "Workbox 'No route found' is benign info, not a bug
  indicator" added at the bottom of
  `.claude/rules/lessons-learned.md` so the next investigation
  doesn't conflate the SW log with a functional break. Pair
  rule "User-perceived bug ≠ code bug: the perception-lag
  class" promotes the broader pattern (verify against
  Network tab + backend state BEFORE patching).

This is the third occurrence in the 2026-05 cycle of
"investigation surfaces a non-bug as the real signal" (the
prior two: D2 verification audit + Multi-tool collaboration
re-sync). Pattern is mature enough that the lessons-learned
section now carries three related rules + a meta-rule on
authoritative-source verification.

### RESTORE-UX-FEEDBACK-01 — optimistic restore + use returned entity

Filed 2026-05-14 by UX-Full-Audit 2026-05-14/15 (Bug A reframe).
P3 IMPROVEMENT. Closed 2026-05-15 in commit `7d68126`.

Context: the 2026-05-14 Articles-Trash Restore user report was
**not a code bug** (workbox "No route found" message was misread
as blocking; the actual restore POST fired correctly). The real
signal was the `[Violation] 'click' handler took 419ms` console
line — perception-lag from chained roundtrips inside a single
synchronous handler. Combined with the "stay in trash view"
pattern post-restore, the feedback was subtle enough that users
reported "doesn't work" even though the backend state was
correct. The audit's "User-perceived bug ≠ code bug: the
perception-lag class" lessons-learned entry captures the
diagnostic pattern.

What landed:

- **`ArticleList.handleRestore` + `Dashboard.handleRestore`
  refactor**: optimistic update — drop the trash row BEFORE
  awaiting the network roundtrip. Use the entity returned by
  POST `/restore` to prepend to the live list. Skip the full
  `/api/articles` or `/api/books` refetch that produced the bulk
  of the lag.
- **Revert on error**: re-add the row to trash state in the catch
  branch so a failed restore doesn't vanish the row entirely.
  Pre-existing silent-failure bug in `ArticleList` (non-ApiError
  exceptions were swallowed) is also closed by the new catch.
- **`Dashboard.handleRestore` signature**: changed from `(id)` to
  `(book)` so the revert path has the entity to push back. Both
  callsites (`TrashCard` + list-view button) updated.
- **i18n**: added `ui.dashboard.restored` +
  `ui.dashboard.restore_failed` across all 8 catalogs (DE, EN,
  ES, FR, EL, PT, TR, JA). Dashboard previously had no success
  toast at all — silent success was itself a UX bug. Added
  English + DE + per-language proper-character translations
  following the catalog's existing transliteration policy.
- **E2E coverage**: new `e2e/smoke/trash-restore-optimistic.spec.ts`
  (4 tests) pins the optimistic-update + revert-on-error
  behavior across both surfaces:
  - "trash row disappears before the restore request resolves"
    (×2, articles + books) — delays the restore via
    `page.route()` and asserts the row is hidden within 500ms.
    Regression-pin: if the handler reverts to "await restore +

  - "trash row is reverted when the restore request fails"

## Archived 2026-05-13 (BACKEND-UPLOAD-SIZE-LIMIT-01)

### BACKEND-UPLOAD-SIZE-LIMIT-01 — body-size cap for every POST/PUT/PATCH

Filed 2026-05-12 by the v0.31.0 dep-audit chain as defense-in-
depth. No specific user report; closed proactively on
2026-05-13 after the BACKUP-HISTORY-SINGLETON-01 close because
the surface fits the same "small, high-value-per-effort, no
external dependency" profile.

Before this fix, only the cover-upload endpoint had a per-
endpoint size guard (~10 MB via the existing
`PayloadTooLargeError`). Every other upload endpoint — Medium
import ZIPs (the frontend caps at 200 MB but the backend was
unbounded), backup .bgb imports, plugin ZIP install, asset
uploads, AI-template ZIP import — would accept arbitrary-size
bodies, exposing a memory-DoS vector.

What landed (1 commit, `5c41d46`):

- `backend/app/middleware/body_size_limit.py`: new ASGI
  `BodySizeLimitMiddleware`. Two enforcement points:
  1. **Content-Length pre-check**: every well-behaved client
     (fetch / XHR / curl POST) sets Content-Length;
     oversized requests return HTTP 413 before any body byte
     is consumed.
  2. **Streamed body counter**: chunked-transfer clients
     (no Content-Length) get a wrapped `receive` that
     accumulates byte counts and aborts with 413 the moment
     the cap is exceeded; wrapped `send` prevents the inner
     handler from leaking a partial response.
  Method allowlist: GET / HEAD / OPTIONS / DELETE bypass
  entirely. Fail-open posture: config-load errors log at
  WARNING but the middleware falls back to the default cap.
- `backend/app/main.py`: middleware registered alongside
  CORSMiddleware; config read once at startup via the
  existing `_load_app_config()` helper.
- `backend/config/app.yaml` + `.example`: new
  `app.max_upload_mb` key (default 500). Editing requires
  a server restart — a runtime cap change would need
  locking around the in-flight request set.
- Per-endpoint caps stay (e.g. cover upload's ~10 MB);
  this is the outer fence, not a replacement.

Tests (+15 in `test_body_size_limit.py`):
- Resolver: 7 tests covering integer / string-int / missing
  / non-positive / non-integer / None paths.
- Integration via TestClient with a 1 KB cap on a single-
  endpoint app: 8 tests covering under-cap pass, at-cap
  pass, over-cap 413, GET/DELETE bypass, JSON-body shape
  for the ApiError contract.
- Full backend suite: 1611 → 1626 (+15, +2 from
  BACKUP-HISTORY-SINGLETON-01 earlier in the session).

Deferred: per-endpoint guards for medium-import and
backup-import. The outer 500 MB middleware covers the DoS
vector; tightening individual endpoints (e.g. backup-import
could refuse > 100 MB outright) is a follow-up if a real
user-flow needs the tighter ceiling. Not filed; reach for
on first user-report basis.

## Archived 2026-05-13 (BACKUP-HISTORY-SINGLETON-01)

### BACKUP-HISTORY-SINGLETON-01 — converge multi-instance backup history through disk

Filed 2026-05-13 alongside the v0.31.0 Phase 2 path-isolation
work. The trigger was active at filing time (the Playwright
smoke spec `import-flows.spec.ts::backup export adds a history
entry` was red); closed on the same day as the next-most-
important backlog item after v0.31.0 shipped.

Root cause: three modules each instantiate their own
`BackupHistory()` at module import time:
- `backend/app/routers/backup.py:24`
- `backend/app/services/backup/backup_export.py:22`
- `backend/app/services/backup/backup_import.py:26`

Before the fix, instance B's `_entries` was set at `__init__`
and never refreshed. The GET `/api/backup/history` endpoint
(served from `routers/backup.py`'s instance) never saw writes
that the export service made through its own instance. A
second, related, multi-worker bug also existed: without
`_load()` in `add()`, a second worker's stale `_entries` list
would clobber the first worker's writes on the next save.

What landed (1 commit, `c90d47d`):

- `backup_history.py`: `list()` and `add()` both call `_load()`
  first. The JSON file is the single source of truth; all
  instances + all uvicorn workers converge through it. Cost:
  one read per call; the file caps at 100 entries (~4 KB
  worst case).
- Module docstring rewritten to document the cross-instance
  consistency contract.
- 2 new regression tests in `test_backup_history.py`:
  - `test_list_sees_writes_from_other_instance_on_same_path` —
    reader-before-writer scenario mirroring the production shape.
  - `test_add_picks_up_writes_from_other_instance` — multi-
    worker write-clobber scenario.

Tests:
- 11 → 13 in `test_backup_history.py` (+2 regression pins)
- Playwright smoke `import-flows.spec.ts`: 7/7 green after the
  Docker dev backend restart that picked up the fix.

Deferred (not filed; reach for if symptoms surface): file-level
OS locks. Not needed for current load (small file, low write
rate); the Python GIL + the uvicorn worker model serialize
operations within a process, and the disk-write race window
across workers is narrow enough that the next reader's
`_load()` will pick up whichever write landed last. No
known production scenario hits this.

### MEDIUM-IMPORT-CONTENT-TRUNCATION-01 — walker section-inner truncation

User reported "einige" (a few) Medium articles imported with
truncated content; pointed at "Wie Deutschland seine Unternehmer
ausbeutet" as the canonical case. Systematic survey of the
209-post production import revealed the actual scope was 117
articles (56%) with content loss, including 9 imports with zero
text. Root cause: `walker._walk_body` called
`section.find("div", class_="section-inner")` which returned
only the FIRST inner div per `<section class="section--body">`.
Medium's standard header-image layout puts THREE inner divs
per section (title, image lane, main body); the walker's
title-skip then dropped the only inner it had picked up, and
the main body was never processed.

Fix in commit `ce9665e`: iterate `find_all` over all inner divs.
Verified against the production worst case: pre-fix 170/3567
words (5%); post-fix 3562/3567 (99.9%).

Retro-fix in commit `18c2031`: re-walk source HTML for affected
articles. Materiality gate skips articles where the new content
is less than 2x the existing AND adds fewer than 100 words.
Image URL preservation: matches recovered ImageRefs against
existing ArticleAsset rows so already-downloaded images keep
their local URLs. Applied to the production corpus: 73 articles
materially recovered; 136 already complete; idempotent re-run
shows zero changes.

### PLUGIN-LOCKFILE-DRIFT-01 — gate the two-installation-paths divergence

Closes the discipline gap surfaced earlier in the same day: the
v0.30.0 release CI red-on-main from plugin lockfile drift after
the pre-v0.30.0 fastapi 0.135 → 0.136 dep sweep. Same shape as
MKDOCS-DISCIPLINE-01 (verify-the-gate); separate item because
the affected installation paths differ.

The shape under discipline: `make test` installs plugins via the
backend's combined `poetry.lock` (path-deps), but CI runs
`poetry install --no-interaction --no-ansi` **per plugin** against
THAT plugin's own `poetry.lock`. When a shared external pin
(e.g. fastapi) bumps in every plugin's pyproject but only the
backend lock gets regenerated, `make test` stays green while CI
turns red. The hotfix `3232fad` re-locked all 10 plugin lockfiles
manually.

Four commits address the gap at three lifecycle layers (commit,
gate-time, post-incident verify):

1. **`1b43aec` feat(make): lock-all-plugins + verify-plugin-locks.**
   Two new Makefile targets (for-loop over `plugins/bibliogon-
   plugin-*/`, per pre-inspection Q1: uniform operation, no
   per-plugin variation that would justify the existing
   per-plugin shortcut convention).
   - `lock-all-plugins`: iterates every plugin and runs
     `poetry lock`. Use after any shared-dep pin bump.
   - `verify-plugin-locks`: runs `poetry install --dry-run
     --no-interaction --no-ansi` per plugin, greps for
     "changed significantly", exits 1 with remediation hint.

## Archived 2026-05-07 (DEP-DBPATH-01 step 3)

### DEP-DBPATH-01 step 3 — BIBLIOGON_DB_PATH override removed

Closes the `BIBLIOGON_DB_PATH` deprecation cycle begun in v0.27.0.
Step 1 (deprecation warning, v0.27.0) and step 2 (precedence flip
making `BIBLIOGON_DATA_DIR` the senior source, v0.28.0) shipped in
prior releases. Step 3 — full removal of the override — lands
between the v0.29.0 tag and the next release tag, accumulated on
`main`.

**Behavior change.** `BIBLIOGON_DB_PATH` is no longer honoured for
path resolution. The resolver in `backend/app/database.py`
short-circuits the env var: if it is set in the environment, a
single warning is logged at startup naming the ignored value, and
path resolution proceeds through the normal `BIBLIOGON_DATA_DIR`
or platformdirs default chain. The warning will be removed in a
later release once existing deployments have migrated.

The new resolution priority is:

1. `BIBLIOGON_TEST=1` → `TEST_DATABASE_URL` or `:memory:`.
2. `DATABASE_URL` (verbatim).
3. `BIBLIOGON_DATA_DIR` derivation → `<DATA_DIR>/bibliogon.db`.
4. Platformdirs default via `app.paths.get_db_path()` (Linux/macOS
   `~/.local/share/bibliogon/`, Windows `%LOCALAPPDATA%\bibliogon\`,
   Docker `/app/data` via the container env-var).

**Tests.** `backend/tests/test_database_url_resolution.py`
rewritten to match the new behavior (6 tests, all green):

- `test_db_path_alone_is_ignored_with_warning` — DB_PATH alone:
  resolver falls through to platformdirs default; warning emitted.
- `test_db_path_with_data_dir_is_ignored_with_warning` — both
  set: DATA_DIR-derived path used; same warning emitted.
- `test_data_dir_alone_does_not_warn` — clean DATA_DIR path; no
  warning.
- `test_neither_env_var_set_does_not_warn` — bare default path;
  no warning.
- `test_database_url_takes_precedence_over_db_path` — DATABASE_URL
  short-circuits including the DB_PATH ignore-warning path.
- `test_test_mode_short_circuits_all_overrides` — BIBLIOGON_TEST
  forces `:memory:` regardless of the other vars.

**Docs.** README.md, README-de.md, and the bilingual deployment
guide (`docs/help/{en,de}/developers/deployment.md`) updated to
mark `BIBLIOGON_DB_PATH` as **Removed in v0.30.0** with the full
deprecation timeline (warning v0.27.0, precedence flip v0.28.0,
removal v0.30.0). The variable row stays in the env-var tables
for one release cycle so users with existing `BIBLIOGON_DB_PATH=`
in their environment can find an explicit "this no longer works"
entry rather than wondering why the doc forgot about it. A later
release drops the row entirely.

**No Docker / launcher / installer impact.** `docker-compose.prod.yml`
sets only `BIBLIOGON_DATA_DIR=/app/data`; no installer script
references DB_PATH. Existing default deployments are unaffected;
only deployments that historically set `BIBLIOGON_DB_PATH` need to
remove it from their environment (the warning at startup names
the value).

**Verification.** `make test` green (backend 1298 + 1 skipped),
ruff + mypy clean, DE content sweep clean (no umlaut
transliterations introduced in the new doc rows).

The next release tag will pick up step 3 along with the post-
v0.29.0 backlog items (docs+i18n session and EL/ES/FR i18n
corrections) and ship as v0.30.0.

### DEP-09 — Vite 7 -> 8 upgrade

Frontend bumped to Vite 8.0.10 + vite-plugin-pwa 1.3.0 + vitest
4.1.5 + @vitest/coverage-v8 4.1.5. Unblocked the same day by
upstream `vite-plugin-pwa@1.3.0` adding Vite 8 to its peer-dep
range (`^3.1.0 || ^4 || ^5 || ^6 || ^7 || ^8`). One Vite 8

between sibling packages (`react` vs `react-dom` vs
`react-router-dom`). Build green: 4 chunks emitted with the

vendor-tiptap ~401 KB, vendor-ui ~218 KB, index ~968 KB),

breaking-change note and the stale-dev-server pollution lesson
caught during this session. Will land in the v0.29.0
CHANGELOG.

### D-05: Auto-install Docker Desktop — closed as won't-fix

Original ROADMAP P4 entry rescoped on 2026-05-04 (commit
`93af0b2`) from "full Windows installer (Docker Desktop + repo +
`.env`)" to "Docker Desktop auto-install only", because the
Bibliogon launcher already handles repo download, `.env`
generation, Docker image build, and browser open on first run
via `_install_or_welcome` / `_run_install_flow` in
`launcher/bibliogon_launcher/__main__.py`. Only Docker Desktop
itself remained terminal-required for fresh-machine setups.

Closed as **won't fix** on 2026-05-05 after the installer
discovery session
(`docs/explorations/installer-discovery-report.md`) confirmed
that the Docker Subscription Service Agreement Section 2.1's
non-sublicensable license grant forbids third-party silent
install of Docker Desktop. Solo-OSS-dev strict-EULA-reading
default applies; no legal-review budget exists to chase an
ambiguous interpretation.

The launcher's existing **detect-and-instruct** flow (open the
official Docker download page when Docker is missing, continue
the install once Docker is present) is the EULA-compliant
ceiling. Replacing it with a third-party silent install would
violate Docker's terms.

**Citation:** see
[docs/explorations/installer-discovery-report.md](../explorations/installer-discovery-report.md)
section "Docker Desktop integration" for the cited EULA
passages and the analysis chain.

**Related but NOT subsumed:** `LAUNCHER-SELFREPLACE-01` remains
an open P4 backlog item. Discovery confirmed it is a separate
scope (atomic launcher binary self-replace on Windows is
unrelated to Docker Desktop installation).
